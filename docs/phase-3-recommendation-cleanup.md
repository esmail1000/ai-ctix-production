# Phase 3 Recommendation Cleanup

This patch keeps the Phase 3 threat-aware risk engine intact and only normalizes the defense recommendation output.

## What changed

- Reported remediation text is no longer split on commas or the word `and`, because remediation sentences often contain comma-separated controls.
- Specific remediation extracted from the report is preferred over generic fallback text such as `Patch the affected component...`.
- Recommendation/source alignment is preserved after de-duplication.
- Threat scenarios now prefer `f.recommendations` generated by the Phase 3 defense engine; graph remediation nodes are used only as fallback.
- Scenario defenses are de-duplicated semantically, not only by exact string match.

## What did not change

- No mock threat intelligence was added.
- No CVSS, CISA KEV, MISP, or known-exploited values are fabricated.
- The risk formula remains deterministic and source-aware.
- The original report severity remains unchanged.

## Acceptance check

Run the same `sample_full_report.txt` through `/api/analyze` and verify:

- SQL Injection recommendations do not split `Use parameterized queries, validate all input...` into broken fragments.
- Apache Struts RCE recommendations start with the extracted `Upgrade Apache Struts...` remediation, not a generic patch message.
- `postAnalysis.threatScenarios[*].recommendedDefenses` does not repeat old graph remediation nodes when risk-engine recommendations exist.
- `riskModelVersion` remains `risk-scoring-v3-threat-aware-deterministic`.
